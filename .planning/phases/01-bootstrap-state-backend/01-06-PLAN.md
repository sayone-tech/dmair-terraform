  <how-to-verify>
    Setup: Open two separate terminal windows/tabs. Both should `cd envs/strapi`.

    Terminal A:
    1. Run: `terraform apply`
    2. Wait until Terraform finishes the refresh/plan phase and prints `Do you want to perform these actions? Enter a value:` (or, if the stack truly has nothing to change, prints `No changes.` and exits — in that case the lock is still acquired during the refresh phase, so the test still works; just be fast in switching to terminal B BEFORE the refresh completes).
    3. **DO NOT** answer the prompt yet. Leave terminal A waiting at the prompt — Terraform holds the S3 state lock for the entire duration of this prompt.

    Terminal B (run while terminal A is still waiting at the prompt):
    1. Run: `terraform apply`
    2. **Expected output (per D-14, RESEARCH §Code Examples §Example 7):** Within a few seconds, terminal B prints:
       `Acquiring state lock. This may take a few moments...`
    3. Terminal B BLOCKS at that message and does not progress.

    (Optional but recommended for ROADMAP SC 4 evidence) Open a third terminal and run:
       `aws --profile dmair s3 ls s3://dmair-terraform-prod/strapi/`
    Expected output includes both `terraform.tfstate` (the state object) AND `terraform.tfstate.tflock` (the sentinel lock object created when terminal A acquired the lock). Capture this output for VERIFICATION.md.

    Cleanup (in this order):
    1. In terminal A: answer `no` (or hit Ctrl-C) at the apply prompt. Terminal A exits and releases the lock.
    2. Terminal B unblocks within seconds, finishes acquiring the lock, runs its own refresh/plan, and reaches its own apply prompt. Answer `no` there too. Both terminals exit without changing any live infra.

    Capture evidence: take a screenshot of both terminals showing terminal B's "Acquiring state lock" message while terminal A is at the apply prompt — OR — capture the full terminal output of both terminals as a text transcript. Save the screenshot to `.planning/phases/01-bootstrap-state-backend/lock-contention-evidence.png` (gitignored if too large; or embed as base64 in VERIFICATION.md) OR paste the transcript into VERIFICATION.md directly.

    If terminal B does NOT print "Acquiring state lock" (e.g., it proceeds straight to plan/apply): BOOTSTRAP-03 has FAILED. Check that `envs/strapi/backend.tf` actually has the `use_lockfile = true` line and that terminal B was started while terminal A was still holding the lock. Re-run.

    Recovery: if either terminal hangs after both apply prompts have been answered, run `cd envs/strapi && terraform force-unlock <LOCK_ID>` using the LOCK_ID printed in the lock-failure message (RESEARCH §Example 7 lock-recovery procedure).
